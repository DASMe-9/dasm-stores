# محرّر ثيمات المتاجر — Store Theme Block Editor

> وثيقة معمارية حيّة. النظير التوضيحي: `store-theme-editor.html`.
> النطاق: تطبيق لوحة تحكم المتجر (`dasm-stores`) — مسار `/dashboard/theme-editor`.
> الحالة: **المرحلة ١ منفّذة** (محرّك البلوكات + الشاشتان + الرسم الفوري + الحفظ). المرحلتان ٢ و٣ مخطّطتان.

---

## ١. الفكرة

محرّر ثيمات على نمط Shopify لأصحاب متاجر داسم. شاشة مقسومة:

- **جهة الكود:** محرّر بلوكات بصيغة وسوم مبسّطة (DSL).
- **جهة المعاينة:** رسم فوري في المتصفح يعكس كل تعديل لحظياً.

أصحاب المتاجر = أصحاب معارض غير مبرمجين. لذلك البلوكات **جاهزة ومعبّأة مسبقاً**؛ المستخدم يحذف/يعدّل نصاً فقط. كل متجر يُولد بثيم افتراضي أنيق (مبدأ «لا متجر فارغ»).

---

## ٢. عقد البيانات — لغة البلوكات (DSL)

وسوم ذاتية الإغلاق، بلوك لكل سطر منطقي:

```
{# الواجهة الافتراضية — عدّل أو احذف أي بلوك #}
<navbar logo links="الرئيسية, المنتجات, تواصل" sticky />
<banner text="شحن مجاني للطلبات فوق ٢٠٠ ر.س" />
<hero title="{{ store.name }}" subtitle="تشكيلة مختارة" cta="تسوّق الآن" style="aurora" />
<featured title="الأكثر مبيعاً" limit="4" />
<product-grid title="كل المنتجات" cols="3" sort="newest" limit="12" />
<footer about="{{ store.name }}" terms="الشروط والأحكام" social="whatsapp, instagram" />
```

**أنواع البلوكات (allowlist):** `navbar` · `banner` · `hero` · `richtext` · `featured` · `product-grid` · `footer`.

**جوهري:** البلوكات **بيانات لا كود**. لا تقييم JS/HTML/Liquid. الرسم = جدول ثابت `BlockType → مكوّن React`. غياب «حوض التنفيذ» يلغي XSS/SSTI من الجذر.

البيانات الديناميكية تُحقن عبر رموز محصورة: `{{ store.name }}` (وتتوسّع لاحقاً). المنتجات تأتي من `{{ store.products }}` داخل `product-grid`/`featured`.

## ٣. خرائط الملفات

### محرّك البلوكات — `lib/themes/blocks/`
| ملف | المسؤولية |
|-----|-----------|
| `types.ts` | أنواع Block/BlockDocument/ParseResult + مفاتيح التخزين |
| `sanitize.ts` | تنقية كل نص: حذف `<script>`/أي وسم/`on*=`/`javascript:` + قصّ الطول |
| `schema.ts` | **حدّ الأمان** — allowlist للأنواع والخصائص + `validateBlock` (إسقاط المجهول + coercion) |
| `parse.ts` | `parseBlocks` (نص→بلوكات) + `serializeBlocks` (بلوكات→نص) — pure |
| `default-template.ts` | الثيم الافتراضي الأنيق «لا متجر فارغ» |
| `store-config.ts` | قراءة/دمج وثيقة البلوكات داخل `theme_config` دون لمس حقول presets |

### الواجهة — `components/theme-editor/`
| ملف | المسؤولية |
|-----|-----------|
| `BlockRenderer.tsx` | الرسم الفوري: switch على نوع البلوك → مكوّنات معاينة (Navbar/Hero/Featured/ProductGrid/Footer…) |
| `SplitEditor.tsx` | الشاشتان: محرّر نص + لوحة إدراج بلوكات + استعادة افتراضي + خانة الذكاء (م٣) + المعاينة |

### الصفحة — `pages/dashboard/theme-editor.tsx`
بوابة توكن → تحميل المتجر → قراءة وثيقة البلوكات من `theme_config` → حالة المصدر → الحفظ يدمج الوثيقة في `theme_config` ويستدعي `sellerApi.updateStore`.

## ٤. التخزين

الوثيقة تُحفظ **إضافياً** داخل `store.theme_config.editor`:

```json
{ "preset_id": "...", "palette": "...",
  "editor": { "version": 1, "source": "<navbar .../>\n<hero .../>" } }
```

لا يكسر اكتشاف presets القديم (`preset_id`/`palette`) — مسار الحفظ نفسه (`PUT /my-store`).

---

## ٥. نموذج الأمان (دفاع متعمّق)

| طبقة | الآلية | الملف |
|------|--------|-------|
| لا مسار تنفيذ | البلوكات وصف لا كود؛ الرسم جدول ثابت | `BlockRenderer.tsx` |
| Allowlist صارم | أنواع/خصائص معروفة فقط؛ الباقي يُسقط | `schema.ts` |
| تنقية النص | حذف الوسوم/المعالجات/`javascript:` | `sanitize.ts` |
| تحقّق قبل الحفظ | `validateBlock` على كل بلوك مُحلّل | `parse.ts` + `schema.ts` |
| عزل النطاق | الواجهة على `stores.dasm.com.sa`، بلا كوكيز مشتركة مع المنصة الأم | بنية النطاقات |
| لا وصول API | الثيم يقرأ `{{ store.* }}` المحقونة فقط | تصميم الرموز |

**أسوأ احتمال:** مطوّر محترف يلصق كوداً خبيثاً ← يُنقّى/يُسقط ← ثيم قبيح في متجره هو فقط. لا يصل المنصة ولا متاجر أخرى ولا الخادم.

## ٦. خطة المراحل

- **م١ (منفّذة):** محرّك البلوكات + الشاشتان + الرسم الفوري + لوحة الإدراج + استعادة الافتراضي + الحفظ.
- **م٢ (منفّذة):** سطحان مستقلان (الواجهة + صفحة المنتجات) مع migration من شكل م١؛ تبويب بصري للسحب/الحذف/الترتيب (`BlockList`) متزامن مع الكود عبر دورة parse→serialize؛ منتجات المتجر الحقيقية في المعاينة (`sellerApi.getProducts`)؛ تبويب «المتجر الفعلي» = iframe على `storefrontUrl(preview)`.
- **م٣ (منفّذة):** مساعد Haiku مدمج — `pages/api/theme/generate.ts` يستدعي `claude-haiku-4-5` بـsystem prompt مُولّد من `describeBlocksForPrompt()` (يبقى متزامناً مع allowlist). **مخرجات النموذج تمرّ بنفس `parseBlocks → serializeBlocks`** فتُنقّى كأي كود مستخدم — لا سطح ثقة جديد. الواجهة: خانة وصف في `SplitEditor` ← `handleGenerate` يستدعي الـroute ويُلحق البلوكات بالسطح النشط. يتطلّب `ANTHROPIC_API_KEY` على Vercel (إن غاب: 503 ورسالة لطيفة). رُفض اللصق الخارجي لأنه يعيد فتح XSS.

### مفتاح البيئة (م٣)
`ANTHROPIC_API_KEY` على مشروع Vercel لـ`dasm-stores`. بدونه يعمل المحرّر كاملاً عدا زر التوليد (يرجع 503).

- **م٤ (منفّذة) — البنّاء البصري:** ١٠ بلوكات جديدة + نوع `url` آمن + ٦ قوالب جاهزة + مكتبة أقسام (`section-library.ts`) + ضوابط تصميم (`ThemeDesign` تحت `theme_config.editor.design`) + علم `hidden` + ٣ أوضاع (مبتدئ افتراضي/ذكاء/متقدّم). **الربط بالواجهة العامة (هجين):** `StorefrontBlocks` يرسم المحتوى عبر `BlockRenderer` والمنتجات عبر `ProductGrid` الحقيقي؛ مبوّب على المتاجر المخصّصة فقط (`lib/storefront-builder.ts`). إصدار الوثيقة ← 3.

### تحديث التخزين (م٢)
الوثيقة صارت سطحين: `theme_config.editor = { version: 2, surfaces: { landing, products } }`. `readBlockDocument` يُرحّل شكل م١ (`{version,source}`) إلى `surfaces.landing` تلقائياً.

---

## ٧. القرارات (محسومة 2026-06-16)

1. لغة الثيم = وسوم مبسّطة مخصّصة (لا Liquid/HTML خام).
2. المعاينة = هجين: رسم فوري بالمتصفح + زر iframe فعلي (م٢).
3. بنية المتجر = سطحان: واجهة أنيقة + شبكة منتجات؛ تفاصيل المنتج قالب محمي.
4. «لا متجر فارغ» = ثيم افتراضي + زر استعادة.
5. الذكاء = مدمج يُخرج بلوكات فقط.
